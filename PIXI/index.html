Features [Link]

PIXI is a sprite insertion tool made to allow more dynamic sprite usage, space savings and more SA-1 compatibility. Despite this, it is almost fully compatible with sprites written for older SpriteTools, save for the fact that Asar is now used as the assembler.

  • Softcoded
    The old sprite tool inserted its code as a preassembled byte array and then sets up the calls (hijacks) for it. Put simply, it was a nightmare to read and figure out what hijacks it applied. Here, the applied asm file is available for editing. The tool only generates the binary tables that are included by the asm. So you can basically look at the code and edit it (to some extent) as you like.
    Additionally, you can create two folders called ExtraHijacks and ExtraDefines that the tool will apply after running all the sprites, thus allowing you to insert arbitrary asm alongside what Pixi already inserts.
  • Dynamic SA-1 support
    The tool's code is 100% SA-1 hybrid. It also provides an asm file with all SA-1 defines that will be default included in every sprite, so you don't have to stick that annoying read1() check for sa-1 at the beginning of every sprite. For all sprite relevant addresses, there exists a define. More info on that is in the readme.
  • GPS Interface
    The user interface is basically the same as p4plus2's GPS for 2 reasons:
    1. It's simple, easy and comfortable to use.
    2. It's a lot more comfortable to work with multiple tools if they all use the same interface.
  • Shared Routines
    Pixi includes (as GPS) a folder listing all the shared routines that can be used in the sprites inserted with the tool. Calling a routine is simply achieved by doing %RoutineName().
    Most of the routines have some included documentation on how to use them correctly in their respective asm files.
    These routines will only be inserted once no matter how many times they're used, and will only be inserted if they're used. This reduces code size by quite a large margin.
  • Asar Only
    Pixi only accepts sprites that are written using Asar syntax. No xkas or TRASM.
  • Sprite Tool support
    The tool basically works the same as sprite tool and might be able to be applied on top of a ROM where Sprite Tool has already been used.
    This process however, is unreliable, as Pixi will attempt to remove all Sprite Tool code beforehand (because of conflicting hijacks) and it may fail at doing so.
    If it succeds however, it will of course remove all the sprites.
  • Per Level Sprites
    Per level sprites are a feature that lets you define a sprite as only "available" only in 1 level and 1 level only. You can have up to 0x10 (16) sprites per each level.
    The 16 slots from B0-BF can be freely assigned across all $200 levels. In the old version of per level sprites, B0-BF were only available as per level slots but the rewrite allows for those slots to be used by normal sprites aswell.
    This way, if you have both a global sprite in slot B0 and some per level sprite in the same slot, Pixi will automatically pick the global one if no per level sprite was defined in that specific level and the opposite otherwise.
    Please note that per level sprites are quite heavy on used ROM space, as they can take up to 2 banks. Likewise, you can also have a shooter, that shoots sprite B0, which will spawn a sprite depending on the level.
    This feature can be particularly useful for collabs
  • CFG Editor
    The CFG Editor that comes with Pixi allows you to freely edit cfg/json files to your liking. It helps with creating custom sprite displays and collections.
